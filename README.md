# Обзор задачи

Устройство хранения данных типа лента (Tape) предназначено для последовательной записи и чтения данных. Считывающая/записывающая магнитная головка неподвижна во время чтения и записи, а лента имеет возможность двигаться в обоих направлениях. Запись и чтение информации возможны в ячейку ленты, на которой в данный момент находится магнитная головка.

Перемещения ленты – затратная по времени операция – лента не предназначена для произвольного доступа.

Имеется входная лента длины N (где N – велико), содержащая элементы типа integer (232). Имеется выходная лента такой же длины. Необходимо записать в выходную ленту отсортированные по возрастанию элементы с входной ленты. Есть ограничение по использованию оперативной памяти – не более M байт (M может быть < N, т.е. загрузить все данные с ленты в оперативную память не получится). Для реализации алгоритма можно использовать разумное количество временных лент, т.е. лент, на которых можно хранить какую-то временную информацию, необходимую в процессе работы алгоритма.

Необходимо сделать следующее:
- Определить интерфейс для работы с устройством типа лента.
- Написать класс, реализующий этот интерфейс и эмулирующий работу с лентой посредством обычного файла. Должно быть возможно сконфигурировать (без перекомпиляции – например, через внешний конфигурационный файл, который будет прочитан на старте приложения) задержки по записи/чтению элемента с ленты, перемотки ленты, и сдвига ленты на одну позицию.
- Файлы временных лент можно сохранять в директорию tmp.
- Написать класс, реализующий алгоритм сортировки данных с входной ленты на выходную.
- Консольное приложение должно принимать на вход имя входного и выходного файлов и производить сортировку.
- Желательно написать юнит-тесты.

Как отправить задание на проверку:
- Решите задачи в своем репозитории на GitHub, вернитесь в тестовое задание, приложите ссылку на ваш репозиторий в текстовое поле справа и отправьте задание
- Перед отправкой убедитесь, что ваш репозиторий содержит все необходимые файлы, вы прикладываете корректную ссылку и по ней есть публичный доступ к вашему решению.

# Общее описание решения

Алгоритм состоит из двух основных этапов:

1. Разбиение входных данных на части, помещающиеся в память
2. Сортировка каждой части и сохранение во временные ленты
3. Слияние всех временных лент

# Архитектура

## Модули

Проект состоит из следующих подсистем:

* config — загрузка и обработка конфигурации
* tape — абстракция ленты и её реализации
* sorter — алгоритм внешней сортировки
* app — приложение
* utils — тут только логирование по итогу

## Модель ленты

Реализован интерфейс ITape, не буду дублировать сюда.

### FileTape

Реализация ленты на основе бинарного файла.

### DelayTapeDecorator

Используется паттерн Декоратор. Добавляет задержки ко всем операциям ленты:

* чтение
* запись
* перемещение
* перемотка

Задержки задаются через структуру конфигурации.

### TempFileTape

RAII-обёртка над FileTape, предназначенная для временных данных:

* генерирует уникальное название файла
* создаёт временный файл
* инициализирует его
* удаляет файл при уничтожении объекта

Используется для хранения промежуточных временных лент.

# Алгоритм сортировки

Реализация находится в компоненте TapeSorter и включает два этапа.

## Временные ленты

Входная лента обрабатывается блоками фиксированного размера:

1. Чтение блока данных, помещающегося в память
2. Сортировка блока в оперативной памяти
3. Запись отсортированного блока во временную ленту
4. Повтор до конца входных данных

Каждый блок формирует отсортированную серию.

## Слияние

Все временные ленты объединяются в одну:

1. Инициализация min-heap (приоритетной очереди)
2. Чтение первого элемента каждой серии
3. Добавление элементов в очередь
4. Итеративное извлечение минимального элемента
5. Запись результата на выходную ленту
6. Движение соответствующей серии вперёд

Сложность слияния: O(N log K)

Простое решение, которое было у меня изначально, работало за O(N * K). Системные тесты на данных большого размера показали, что разница во времени работы двух алгоритмов существенная!

# Конфигурация

Система настраивается через текстовый конфигурационный файл формата key=value.

Пример:

```
// комментарий
tape.read_delay_ms    = 1
tape.write_delay_ms   = 1
tape.move_delay_ms    = 1   // еще комментарий для примера
tape.rewind_delay_ms  = 3

memory.limit_bytes    = 40

filesystem.tmp_dir    = ./tmp
```

Поддерживаются:
* комментарии //
* пробелы вокруг ключей и значений

Параметры:
* tape.* — задержки операций ленты
* memory.limit_bytes — ограничение памяти для сортировки
* filesystem.tmp_dir — директория временных файлов

# Тестирование

Проект использует GoogleTest.

## Юнит-тесты

В проекте покрыты следующие классы:

* ConfigLoader — парсинг конфигурационного файла, обработка комментариев, пробелов, ошибок формата и отсутствующих ключей.
* DelayTapeDecorator — корректная делегация операций в ITape и применение задержек для каждой операции.
* FileTape — работа с файловой лентой: чтение, запись, перемещение головки, обработка границ (BOF/EOF) и ошибок.
* TempFileTape — создание и удаление временных файлов, инициализация данных, корректная работа позиционирования и операций записи/чтения.

## Интеграционные тесты

Интеграционные тесты проверяют работу всей системы целиком в режиме black-box через CLI-интерфейс приложения (`App::run`).

Тестируется полный pipeline обработки данных:

* разбор CLI аргументов
* загрузка конфигурации
* инициализация tape-слоя
* выполнение внешней сортировки
* запись результата в выходной файл

## Нагрузочные и граничные случаи

Отдельный набор тестов проверяет корректность работы системы на крайних и стрессовых сценариях:

* пустой входной файл
* один элемент
* уже отсортированные данные
* данные в обратном порядке
* наличие дубликатов
* отрицательные значения
* случайные массивы (проверка корректности сортировки)
* большие объёмы данных (до 50 000 элементов)
* экстремально малый лимит памяти (1 элемент в памяти)

Эти тесты проверяют устойчивость реализации внешней сортировки и корректность работы при ограничениях памяти и интенсивных IO-операциях.
